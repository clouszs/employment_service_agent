# 阶段 10 进度追踪：多智能体触发评估

> **目标**：按检查点清单正式评估是否需要引入 Supervisor + Subgraph
> **前置条件**：阶段 9 完成（单智能体扩展期）
> **评估日期**：2026-06-27
> **评估结论**：✅ 无需引入，继续单 Agent 架构

---

## 评估背景

阶段 9 完成后，系统具备以下特征：

| 指标 | 数值 |
|------|------|
| Agent 工具数 | 8 个（触达阈值边缘） |
| 意图分类 | 9 类 |
| 工作流节点数 | 12 个 |
| 工作流结构 | 单流程，无独立子流程 |
| 状态管理 | 扁平 `AgentState` |
| LLM 模型 | 全部通义千问 |
| 单次 invoke LLM 调用 | 1-2 次 |

对照 [多智能体架构设计文档](../implementation/multi-agent-architecture.md) 的决策矩阵，逐条评估。

## 检查点清单

### 检查点 1：出现 3+ 个独立子流程

| 项目 | 内容 |
|------|------|
| **阈值** | 3 个及以上独立子流程 |
| **当前状态** | 单流程 12 节点，所有节点共享同一工作流 |
| **评估结果** | ❌ 未触发 |
| **说明** | 即使意图分类扩展为 9 类，仍通过同一 `route_query` 节点分流，各意图共享后续检索→生成→防御流程 |

### 检查点 2：Agent 间需要状态隔离

| 项目 | 内容 |
|------|------|
| **阈值** | 需要隔离的上下文（如求职 Agent 不能访问对话历史） |
| **当前状态** | `AgentState` 扁平结构足够，所有字段可选读写 |
| **评估结果** | ❌ 未触发 |
| **说明** | 各工具（resume/job/calendar）通过 `AgentState` 的不同字段隔离数据，无需独立状态图 |

### 检查点 3：不同 Agent 需要不同模型

| 项目 | 内容 |
|------|------|
| **阈值** | 不同业务域需要不同 LLM 模型/提示词策略 |
| **当前状态** | 全部使用通义千问，提示词通过工具参数适配 |
| **评估结果** | ❌ 未触发 |
| **说明** | 各意图的 Prompt 差异可通过 `route_query` 分支 + 工具参数差异化实现，不需要独立 Agent |

### 检查点 4：工具数量 > 8 个

| 项目 | 内容 |
|------|------|
| **阈值** | > 8 个工具导致单 Prompt 超过 4k tokens |
| **当前状态** | **恰好 8 个**（等于阈值，未超过） |
| **评估结果** | ❌ 未触发（边缘状态） |
| **工具清单** | `knowledge_search` / `bing_search` / `fetch_webpage` / `toggle_faq_status` / `generate_resume` / `recommend_jobs` / `add_calendar_event` / `fetch_announcement` |
| **风险说明** | 当前卡在阈值边缘，阶段 11 开发中新增工具必须谨慎 |

### 检查点 5：单 Prompt 超过 4k tokens

| 项目 | 内容 |
|------|------|
| **阈值** | 单 Prompt > 4k tokens |
| **当前状态** | 提示词精简，未超过 4k tokens |
| **评估结果** | ❌ 未触发 |
| **说明** | 当前工具描述 + 系统 Prompt 合计在合理范围内 |

## 触发信号监控清单

阶段 11 开发中，以下任一信号出现时必须**立即暂停**并重新评估：

| # | 触发信号 | 应对措施 |
|---|----------|----------|
| 1 | 工具数超过 8 个 | 立即评估是否引入 Subgraph |
| 2 | 简历助手需要"生成 → 预览 → 修改"的多轮独立流程 | 引入 `ResumeSubgraph` |
| 3 | 求职功能和问答功能的 Prompt/防御逻辑出现显著差异 | 考虑 `JobAgent` Subgraph |
| 4 | 单次 `invoke` 中出现 3+ 次串行 LLM 调用导致超时 | 将 LLM 密集流程拆为 Subgraph |

## 阶段 11 工具扩展策略

### 原则：封顶 8 个工具

阶段 11 开发新业务模块（企业/职位/招聘会/简历/面试/就业数据）时：

1. **优先复用现有工具**：通过修改工具参数适配新场景
2. **其次合并同类工具**：将相似功能合并为一个工具的扩展参数
3. **最后才考虑新增工具**：新增前必须完成本评估报告的升级版

### 工具扩展映射建议

| 新业务模块 | 复用工具 | 适配方式 |
|-----------|---------|---------|
| 企业管理 | `knowledge_search` | 增加企业类文档到知识库 |
| 职位推荐 | `recommend_jobs` | 已实现，直接复用 |
| 招聘会 | `fetch_announcement` | 扩展参数支持招聘会类型筛选 |
| 面试日程 | `add_calendar_event` | 已实现，直接复用 |
| 就业数据 | REST API | 不走 Agent 工具 |
| 简历助手 | `generate_resume` | 已实现，V1 返回 JSON |
| 公告管理 | `fetch_announcement` | 已实现，直接复用 |
| 系统设置 | REST API | 不走 Agent 工具 |
